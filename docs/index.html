Measuring congestion-aware routing in GPU fabrics

A deterministic discrete-event simulator and three routers that plug into it.
Source on GitHub · architecture diagram · last updated 2026-07-18

Abstract. Collective traffic on GPU training fabrics produces transient congestion (incast bursts, ECMP hash collisions) that static routing cannot react to, and which shows up almost entirely in the tail of the flow-completion-time (FCT) distribution. This project builds a small, deterministic fabric simulator (C++20, integer-picosecond virtual clock, packet-train granularity), validates it against the published incast throughput-collapse result before trusting it, and then compares three routers behind one interface. A CONGA-style flowlet router halves p99 FCT against ECMP on permutation traffic over an oversubscribed Clos. A reinforcement-learning router, given the same telemetry, loses to both baselines; the diagnosis (§5) is more useful than a win would have been. Same seed, same config, same bytes, every time.

1. Why bother

When a thousand GPUs run all-reduce, the network is quiet, then everyone talks at once, then it is quiet again. Averages hide this completely. The damage is done by the worst few flows per iteration — the p99 and p99.9 stragglers that every other GPU waits for. So the question worth asking is narrow: how much of the tail can routing actually recover? Not "can we build a smarter router" in the abstract, but a measured delta against the boring control, on stated traffic, with error bars.

Simulation is the honest way to ask it, provided the simulator itself is trustworthy. Most aren't, in a specific way: they are validated against nothing, and then their router comparisons inherit the vacuum. The rule here was simple: no router work of any kind until the simulator reproduced a known published result (§3).

2. What was built

A single-threaded discrete-event engine. The virtual clock is an int64 counting picoseconds, which makes serialization delays exact for whole-Gbps link rates and removes floating-point drift from event ordering entirely.1 Events are scheduled per chunk (64 KiB packet train), not per packet — fine enough that queues, drops, and ECN marks emerge; coarse enough that a 1024-host all-reduce (2.1 million flows, 17 million events) finishes in 5.5 seconds on one core.

On top of that: two-tier Clos topologies; per-link FIFO queues with buffer limits, tail-drop, and ECN threshold marking; a deliberately TCP-shaped transport (windowed, SACK-style fast retransmit, RTO with exponential backoff); and workloads for incast, ring all-reduce, and random permutation. Everything that consumes randomness draws from a named stream derived from one master seed, and every stream is logged per run. Two runs with the same seed and config produce byte-identical output, and a CI job holds the project to that.

Routing is one interface. A router sees fabric telemetry (live queue depths, drop and mark counters, a utilization average) and returns a path. It never sees the wall clock. Three implementations sit behind it:

RouterOne-line descriptionRole
ECMPHash the flow id, pick a spine, never look backcontrol
FlowletLeast-congested spine, with stickiness and hysteresisthe heuristic to beat
RLTabular bandit; reward is negative queueing delaythe experiment

3. Earning the right to compare

The gate: drive a synchronized many-to-one incast pattern at a shallow-buffered bottleneck and check that the simulator produces the throughput collapse reported by Vasudevan et al. at SIGCOMM 2009. It does: goodput climbs to 0.98 of line rate at 8 senders, falls off a cliff past 10, and grinds down to 0.08 at 63:

Getting the curve's shape right forced two pieces of transport realism: without fast retransmit, a single tail-drop costs a full RTO and the curve collapses at every sender count; without a window cap, additive increase overflows the buffer at every sender count. Both are the kind of bug a pretty but unvalidated simulator would have carried silently into every router comparison downstream.

4. Results

Setup: 64-host two-tier Clos, four spines, 4:1 oversubscription at the leaf; every host sends 2 MiB to a random partner (a seeded derangement); 20 seeds, and the same 20 traffic patterns are replayed for every router, so comparisons are paired. Percentiles are pooled with bootstrap confidence intervals over seeds.

The headline is the middle row: the flowlet router halves pooled p99 FCT and leaves the median alone, which is exactly the signature you want from adaptive routing; it should only matter when hashing gets unlucky.

It did not start out that way. The first version of the adaptive router simply sent every flowlet to the least-queued spine, and it measured eight times worse than ECMP.2 Queue telemetry lags a send decision by the host-to-leaf latency, so every sender saw the same "empty" spine, dogpiled onto it, then dogpiled off it in unison. The fix was not better information but coordination: a flow keeps its current path unless an alternative beats it by more than the flow's own backlog, and ties break by a salted hash so simultaneous deciders spread out. Herding, hysteresis, decorrelation — remember those three words for §5.

5. The router that lost

The RL router is a tabular contextual bandit. Its state is the pair of leaves plus each candidate spine's congestion bucket, built from the same telemetry snapshot the flowlet router reads — information parity was a design requirement, so that any gap is attributable to the policy rather than the inputs. Reward is negative queueing delay, fed back per delivered chunk. Action selection is ε-greedy from a seeded, logged stream. All of it deterministic and unit-tested: on a stationary bottleneck it reliably learns to route around the congested spine.

On the real workload it loses to both baselines. Exploration is not the reason; that only explains the doubled median. The damning observation is what happened when exploration was reduced: p99 got worse, not better, doubling to 31.8 ms. A greedier shared policy means more agents taking the same argmax at the same moment on the same stale state. The learned router herds structurally, the exact pathology of §4 — but where the heuristic carries stickiness and hashed tie-breaking as built-in decorrelation, the bandit has nothing pulling concurrent decisions apart.

So the honest conclusion is not "RL doesn't work." It is narrower and more useful: a per-decision learned policy, shared by many simultaneous actors on lagging state, needs a coordination mechanism before it can even match a well-tuned heuristic. And within a single 16-millisecond episode there are only about two thousand decisions spread over thousands of table states, so it is data-starved on top of it. Cross-episode training and per-flow decorrelation are the obvious next experiments, and the harness is already built for them.

6. Reproducing everything

Each figure on this page regenerates from one command. Runs land in a directory stamped with the config hash and git SHA; the manifest records whether the tree was dirty, and a dirty run is by definition not publishable.

# build (C++20, no third-party dependencies in the core)
cmake -B build -DCMAKE_BUILD_TYPE=Release && cmake --build build -j
python3 -m venv .venv && .venv/bin/pip install -r requirements.txt

# figure 1 — exits nonzero if the collapse fails to reproduce
.venv/bin/python python/validate_incast.py

# figure 2 and the table — 20 paired seeds, three routers
.venv/bin/python python/harness.py --routers ecmp,flowlet,rl --seeds 20

# figure 3 — from any run directory containing links.parquet
.venv/bin/python python/plot_heatmap.py results/sweep/runs/<run>

Tests run under AddressSanitizer and UBSan with warnings-as-errors on every push, including an analytic anchor: a single uncontended flow's completion time must equal the closed-form store-and-forward value, exactly, to the picosecond.

7. Limitations

Stated plainly, because they are the boundary of the claim. Chunk granularity, not per-packet; no PFC or lossless fabrics; single failure domain; static topology. The transport's ECN response is a blunt once-per-RTT halving rather than DCTCP's proportional response, which is why the validation gate rests on the tail-drop curve. Under any rerouting scheme, p99.9 can regress on a minority of seeds (2 of 20 here) when a rerouted burst lands in an already-full queue and climbs the RTO backoff ladder. Headline experiments ran at 64 hosts; the engine is exercised to 1,024. Nothing here says "hundreds of thousands of GPUs," because nothing here measured that.
